<objective>
Fix the missing initial batch-progress event emission so the overall progress bar shows the correct file count ("0/n") from the start of processing, not "0/1" during the first file.

Two root causes: (1) Rust `start_batch` initializes `BatchProgress` with the correct total but never emits it; (2) Frontend `useBatch.startBatch()` hardcodes `startProcessing(1)` as a workaround. This plan fixes both.

Purpose: The overall batch progress bar must show an accurate total file count throughout processing. The "percentage complete" requirement of ROADMAP SC-1 (BATCH-02) is partially broken when total displays as 1 during the first file.
Output: Rust emits initial batch-progress event. Frontend passes actual queue size to startProcessing before invoke.
</objective>
