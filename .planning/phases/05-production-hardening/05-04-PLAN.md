  <action>
Per D-07: reduce worker thread idle time by removing unnecessary synchronization barriers. The current batch loop acquires the `state` Mutex and `batch_state` Mutex independently for EACH status update (current_file, succeeded, failed, completed). This creates four lock acquisitions per file. Optimization: combine progress field updates into a single lock acquisition.

In `src-tauri/src/commands/batch.rs`, modify the per-file processing block (lines 172-235):

**Before (pseudocode of current pattern):**
```
for each file:
    lock state -> lock batch_state -> update current_file  // acquisition 1
    execute_single_file(...)
    if Ok:
        lock state -> lock batch_state -> update succeeded, completed  // acquisition 2
    if Err:
        lock state -> lock batch_state -> update failed, completed  // acquisition 3
    lock state -> lock batch_state -> emit progress  // acquisition 4
```

**After (optimized):**
Replace the separate lock acquisitions in the `Ok` and `Err` branches with a single acquisition that updates all fields at once:

Find the `Ok(output_path)` arm (lines 191-199) and replace the block from that point through the end of the processing loop (line 234) with this optimized version:

```rust
            match execute_single_file(&app, entry, &seed, &ffmpeg_dir, &output_dir, &cancel_flag, gpu_encoder.as_ref()) {
                Ok(output_path) => {
                    succeeded_files.push(output_path);
                    // D-07: Single lock acquisition for all progress updates
                    let app_state = state.lock().map_err(|e| format!("Lock error: {}", e))?;
                    let mut batch_state = app_state
                        .batch_state
                        .lock()
                        .map_err(|e| format!("Batch state lock error: {}", e))?;
                    batch_state.progress.succeeded += 1;
                    batch_state.progress.completed += 1;
                    let progress_snapshot = batch_state.progress.clone();
                    drop(batch_state);
                    drop(app_state);
                    let _ = app.emit("batch-progress", progress_snapshot);
                }
                Err(e) => {
                    let _ = app.emit(
                        "batch-file-error",
                        FileResult {
                            file: entry.filename.clone(),
                            seed: seed.alias.clone(),
                            error: e.clone(),
                        },
                    );
                    failed_files.push(FileResult {
                        file: entry.filename.clone(),
                        seed: seed.alias.clone(),
                        error: e,
                    });
                    // D-07: Single lock acquisition for all progress updates
                    let app_state = state.lock().map_err(|e| format!("Lock error: {}", e))?;
                    let mut batch_state = app_state
                        .batch_state
                        .lock()
                        .map_err(|e| format!("Batch state lock error: {}", e))?;
                    batch_state.progress.failed += 1;
                    batch_state.progress.completed += 1;
                    let progress_snapshot = batch_state.progress.clone();
                    drop(batch_state);
                    drop(app_state);
                    let _ = app.emit("batch-progress", progress_snapshot);
                }
            }
```

Key changes:
1. Each branch acquires locks ONCE (not twice: once for update, once for emit)
2. Clone `BatchProgress` while lock is held (avoids deadlock from `app.emit` inside lock scope)
3. Explicit `drop()` before `app.emit()` to release locks before event emission (event listeners may try to acquire `state` lock)
4. Removes the separate progress emission block that followed the match (lines 226-234 in current code) — emission is now inline in each branch

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| FFmpeg process -> executor progress events | Untrusted FFmpeg stderr parsed by ffmpeg-sidecar |
| batch.rs AppState lock -> frontend event listeners | Events emitted after lock release to prevent deadlock |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-04-01 | Denial of Service | Mutex deadlock in batch.rs | mitigate | `drop(batch_state)` and `drop(app_state)` before `app.emit()` ensures locks are released before event listeners execute. Event listeners may call `get_batch_status()` which re-acquires the state lock — no deadlock because the previous lock is dropped. |
| T-05-04-02 | Tampering | batch-progress event payload | accept | `BatchProgress` is cloned from authoritative state while the lock is held. No stale/inconsistent data emitted. Serialization via serde is automatic. |
| T-05-04-03 | Denial of Service | Streaming I/O pipe buffer deadlock | accept | executor.rs uses `child.iter()` (ffmpeg-sidecar 2.5.x) which drains stderr continuously. The `for event in child.iter()` pattern prevents pipe buffer deadlock. Verified: this is the standard ffmpeg-sidecar API. |
</threat_model>
