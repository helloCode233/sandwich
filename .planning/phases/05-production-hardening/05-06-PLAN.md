```rust
/// Compute MD5 hash of a file via streaming I/O (Phase 5: MD5-01, MD5-02).
/// Uses an 8KB buffer to avoid loading the entire file into memory.
/// Per RESEARCH Pitfall 4: always use streaming I/O; never load entire file for MD5.
pub fn file_md5(path: &Path) -> Result<String, String> {
    use md5::Md5;
    let mut file = std::fs::File::open(path)
        .map_err(|e| format!("Cannot open file for MD5: {}", e))?;
    let mut hasher = Md5::new();
    let mut buf = [0u8; 8192];
    loop {
        let n = file
            .read(&mut buf)
            .map_err(|e| format!("MD5 read error: {}", e))?;
        if n == 0 {
            break;
        }
        hasher.update(&buf[..n]);
    }
    Ok(format!("{:x}", hasher.finalize()))
}
```

Per RESEARCH.md: uses `md5` crate (not `fast-md5`), streaming I/O with 8KB buffer (not `md5::compute(&[u8])` which loads entire file), returns hex string. Per Pitfall 4: wrap in `tokio::task::spawn_blocking()` at call site (done in Task 2 batch.rs changes).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| File system -> MD5 hash function | File contents read via streaming I/O into md5::Md5 hasher |
| Rust BatchResult -> Tauri IPC -> frontend | MD5 hex strings serialized in FileSuccess over event system |
| BatchSummary UI -> user display | MD5 hash values rendered as text in the DOM |
| FFmpeg GPU process -> retry path | Failed GPU encode triggers re-spawn with CPU encoder |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-06-01 | Denial of Service | MD5 of large files blocking async runtime | mitigate | `tokio::task::spawn_blocking()` wraps all file I/O and MD5 computation. Pre-processing hashes run one-at-a-time (sequential) to avoid disk thrashing; post-processing runs per-output. GPU retry path also uses spawn_blocking for post-retry MD5. RESEARCH Pitfall 4 explicitly addresses this. |
| T-05-06-02 | Information Disclosure | MD5 hash values in frontend events | accept | MD5 is a one-way hash. Even if the full 32-char hex is displayed, it reveals nothing about the file contents. This is a desktop app — no network transmission of hashes. |
| T-05-06-03 | Tampering | MD5 comparison logic | accept | Comparison is `md5_before != md5_after`. Standard Rust string equality. No cryptographic verification needed — MD5 is used for change detection only, per ASVS V6 advisory in RESEARCH.md. |
| T-05-06-04 | Repudiation | Unchanged MD5 (processing had no effect) | mitigate | `modified: false` flag + amber warning icon in UI + warning banner with count. User is informed when processing was ineffective. No silent failure. |
| T-05-06-05 | Denial of Service | Disk I/O for MD5 pre-hash on many large files | accept | Pre-hashing is sequential and occurs before the processing loop. Large queues (many files) will take longer to start processing, but this is inherent to integrity verification. User sees no progress bar movement during pre-hash phase — addressed in documentation. |
| T-05-06-06 | Denial of Service | GPU encoder hang during retry | mitigate | Per D-05: GPU encode failure triggers exactly one retry with CPU (libx264). The retry uses `gpu_encoder: None` which skips GPU injection entirely. If CPU retry also fails, the file is permanently failed with combined error message. No infinite retry loop. The existing cancel flag is checked between retry attempts. |
| T-05-06-07 | Elevation of Privilege | GPU retry path arg injection | accept | The retry call passes `gpu_encoder: None` which causes the executor to use the hardcoded `"libx264"` fallback. No user-supplied strings reach the retry path's encoder selection. |
