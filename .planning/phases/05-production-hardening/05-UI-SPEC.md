## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 (regular) | 1.5 |
| Label | 12px | 400 (regular) | 1.5 |
| Heading | 16px | 600 (semibold) | 1.2 |
| Mono (MD5) | 12px | 400 (regular) | 1.4 |
| Detail | 11px | 400 (regular) | 1.4 |

Implementation:
- Body: `NText` with `class="text-sm"` (14px, regular weight)
- Label: `NText` with `class="text-xs"` and `depth="2"` or `depth="3"` (12px)
- Heading: `NText strong class="text-base"` (16px, semibold)
- MD5 hash values: `NText depth="3" class="text-xs font-mono"` (12px, monospace for hex readability). Use UnoCSS `font-mono` utility.
- Detail (seed alias, MD5 short values, output path in summary rows): `NText depth="3" class="text-[11px]"` (11px, regular). Used for secondary metadata in dense summary card layouts where 12px would reduce information density too much. Uno arbitrary value `text-[11px]`.

Weight tokens: **400 (regular)** and **600 (semibold)**. No other weights used.

Source: Pre-populated from existing `BatchControls.vue` (text-xs for hints, text-sm for labels, text-base for headings), `BatchBanner.vue` (text-xs for progress labels), `BatchSummary.vue` (text-base for title, text-xs for rows). Mono font for MD5 hex is a Phase 05 addition.

## Visual Hierarchy

**Focal point:** The BatchBanner progress bar (animated NProgress with `#2080f0` fill) is the primary visual anchor during active processing. Its color + motion draw the eye first.

**Eye path during processing:**
1. **BatchBanner progress bar** — motion + accent color (#2080f0), anchors the top of the panel
2. **Per-file progress indicators** in QueueList — green (#18a058) NProgress bars, one per active file
3. **Multi-seed selector** (BatchControls NSelect) — disabled during processing, visually recessed
4. **Cancel button** — error-colored (#d03050), always visible as the escape hatch

**Eye path post-completion:**
1. **BatchSummary status icons** — color-coded (green CheckCircle / amber ShieldAlert / red XCircle) draw attention to results
2. **MD5 comparison rows** — monospace hex values create visual rhythm; amber "Unchanged" labels stand out as anomalies
3. **Summary body text** — provides numerical recap (succeeded/failed/unchanged counts)
4. **Clear Results button** — secondary action, bottom-right, visually recessive

**Hierarchy during idle state (unchanged from Phase 04):**
1. Seed list (left panel) — compact cards with hover-reveal actions
2. Import zone + video queue (right panel top) — dashed-border drop zone
3. Batch controls (right panel bottom) — seed selector, concurrency, output dir, start button

### seedStore Changes

| Field | Before (Phase 04) | After (Phase 05) |
|-------|-------------------|------------------|
| `selectedSeedId` | `string \| null` | Removed (or kept as deprecated derived from first selected) |

Implementation note: `selectSeed(null)` is removed. Replace with `toggleSeed(id)` that toggles membership in the `selectedSeedIds` array. Clearing selection is done via `selectedSeedIds = []` or `toggleSeed` on the already-selected item.

### batchStore Changes

| Field | Before (Phase 05 start) | After (Phase 05) |
|-------|------------------------|-------------------|
| `progress.total` | `number` (file count) | `number` (file count x seed count) |

| `lastResult.failed` | `FileResult[]` | Unchanged |

### Type Changes (`src/types/batch.ts`)

New types:

```typescript
export interface MDSucceededFile {
  path: string;         // output file path
  inputPath: string;    // original input file path
  md5Before: string;    // MD5 of input file before processing
  md5After: string;     // MD5 of output file after processing
  modified: boolean;    // md5Before !== md5After
  seedAlias: string;    // which seed produced this output
}

export interface BatchResult {
  succeeded: MDSucceededFile[];  // was: string[]
  failed: FileResult[];          // unchanged
}

export interface PerFileProgress {
  file: string;
  seedAlias: string;             // NEW: which seed is being applied
  percent: number;
  currentFrame: number;
  totalFrames: number;
  fps: number;
  remainingSeconds: number;
}
```

Source: CONTEXT.md D-12, D-13, D-14, D-15, Claude's Discretion items.

### useBatch Composable Changes

```typescript
// Before (Phase 04):
async function startBatch(seedId: string, outputDir: string, queueSize: number): Promise<boolean>

// After (Phase 05):
async function startBatch(seedIds: string[], outputDir: string, queueSize: number): Promise<boolean>
```

The `queueSize` parameter becomes the raw file count. The composable computes `total = queueSize * seedIds.length` for `store.startProcessing(total)`.

Source: CONTEXT.md D-10, D-12.

## Interaction Contract

### Multi-Seed Selection Flow

1. User clicks NSelect dropdown
2. Dropdown shows all seeds with checkboxes (NSelect `multiple` mode)
3. User checks 2+ seeds. Each checked seed appears as a tag inside the selector.
4. When dropdown closes, selected count is visible in the closed selector: "2 seeds selected" (via `max-tag-count` collapsing)
5. Start button enables when >= 1 seed selected AND >= 1 valid queue item
6. Processing starts: for each file, for each seed, execute processing

**Edge case:** If user deselects all seeds during processing (not possible -- NSelect is `:disabled="batchStore.isProcessing"`), no action. Prevention: selector disabled during processing.

### MD5 Summary Interaction

1. Batch completes. BatchBanner shows "Batch Complete".
2. BatchSummary renders below with succeeded/failed/unchanged sections.
3. MD5 columns show shortened hash (first 8 chars). User hovers to see `title` tooltip with full 32-char hash.
4. Files with unchanged MD5 have amber warning icon + label.
5. Unchanged warning block appears below all file rows if any files are unchanged.
6. User clicks "Clear Results" -- summary disappears, batch resets to idle.

**Edge case:** All files have unchanged MD5. All rows appear in succeeded section with amber warning icons. Warning block says "N file(s) output MD5 matches input -- processing may have had no effect."

### GPU Notification Flow

1. App/backend detects GPU encoder on first batch start.
2. If GPU encoder found: no UI notification (silent success). Encoder name logged to console only.
3. If GPU encoder unavailable: `useMessage.info()` notification appears before first file: "GPU encoder unavailable, using CPU (libx264)" / "GPU 编码器不可用，使用 CPU（libx264）". Auto-dismiss after 5 seconds.
4. No persistent GPU status indicator. No settings UI for encoder selection (per D-06).

Source: CONTEXT.md D-04, D-05, D-06.
